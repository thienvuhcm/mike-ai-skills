# Java Code Intelligence with JDTLS

> **TL;DR** — When working in Java, use the **LSP tool** (backed by the Eclipse JDT Language Server, JDTLS) instead of grep/view/sed for anything that has to do with *symbols*. It's faster, more accurate, and understands imports, generics, and inheritance.

## Why JDTLS

[Eclipse JDT Language Server](https://github.com/eclipse-jdtls/eclipse.jdt.ls) is the same Java engine that powers VS Code's Java extension, IntelliJ's LSP mode, Eclipse Theia, Neovim's `nvim-jdtls`, and GitHub Copilot CLI's `lsp` tool. It reads `pom.xml` directly — no extra configuration needed — and provides:

- Symbol navigation (`goToDefinition`, `goToImplementation`)
- Usage analysis (`findReferences`, `incomingCalls`, `outgoingCalls`)
- Semantic rename across files
- Hover with Javadoc, type info, and parameter hints
- Real-time diagnostics (compile errors, unused imports, null-check issues)
- Workspace-wide symbol search

For an AI agent, this means: **verify a method exists before calling it, rename safely, catch compile errors before `./mvnw verify`**. Much cheaper in tokens and latency than generating code that fails at build time.

## Installing JDTLS

| Platform | Command |
|---|---|
| macOS (Homebrew) | `brew install jdtls` |
| Linux (Homebrew on Linux) | `brew install jdtls` |
| Linux (manual) | Download release from [eclipse-jdtls/eclipse.jdt.ls releases](https://github.com/eclipse-jdtls/eclipse.jdt.ls/releases), extract, and put the launcher on `PATH` |
| Windows | Use WSL + `brew install jdtls`, or download the release and add to `PATH` |
| mise / asdf | `mise use jdtls@latest` |

Prerequisite: **Java 17+** on `PATH` (we require Java 25 anyway — see [`versions.json`](../versions.json)).

Verify the install:

```bash
jdtls --help
```

## Wiring JDTLS into your agent

### GitHub Copilot CLI

Copilot CLI reads LSP configuration from:

- **Repository-level**: `.github/lsp.json` — generated by this skill automatically for every new project (see [`assets/lsp.json`](../assets/lsp.json)).
- **User-level**: `~/.copilot/lsp-config.json`.

The template this skill ships includes performance tweaks that match the recommended Copilot CLI user-level config:

```json
{
  "lspServers": {
    "java": {
      "command": "jdtls",
      "args": [
        "--jvm-arg=-Xmx4G",
        "--jvm-arg=-Dlog.level=WARN",
        "--jvm-arg=-XX:+UseStringDeduplication",
        "-data", ".jdtls-workspace"
      ],
      "fileExtensions": {
        ".java": "java"
      }
    }
  }
}
```

Why these flags:

- `-Xmx4G` — gives JDTLS enough heap to index a full Spring Boot project without GC thrash.
- `-Dlog.level=WARN` — silences verbose INFO logs that otherwise spam the session.
- `-XX:+UseStringDeduplication` — lowers retained heap (JDTLS keeps a lot of interned strings).
- `-data .jdtls-workspace` — stores the JDTLS index **per project** (relative to the repo root) instead of a single shared cache. This prevents cross-project classpath contamination and lets you nuke a stale index with `rm -rf .jdtls-workspace`. The generated `.gitignore` excludes this directory.

Check the wired server inside a Copilot CLI session:

```
/lsp
```

- **Neovim**: [`nvim-jdtls`](https://github.com/mfussenegger/nvim-jdtls)

## Verifying it works

1. **In Copilot CLI**: run `/lsp` — you should see `java` listed as configured.
2. **In VS Code**: look for `Java: Ready` in the status bar (takes 30–60s on first open of a large project while JDTLS indexes).
3. **Smoke test with the agent**: ask it to "find all usages of `Application.class`". If it uses grep instead of `findReferences`, JDTLS isn't wired — check the binary on `PATH` and the config path.

## When to use LSP vs grep/view

| Task | Right tool |
|---|---|
| "Where is `UserRepository` defined?" | `lsp goToDefinition` |
| "Who calls `save(User)`?" | `lsp findReferences` or `incomingCalls` |
| "What does `@Transactional(readOnly = true)` do?" | `lsp hover` |
| "List methods in `UserService.java`" | `lsp documentSymbol` |
| "Find any class matching `*Controller`" | `lsp workspaceSymbol` |
| "Rename `getUser()` to `findUser()` across the project" | `lsp rename` (never sed) |
| "Are there compile errors right now?" | `ide-get_diagnostics` (VS Code IDE tool) |
| Read a text file, YAML, properties | `view` / `grep` |
| Search a non-symbol string (log message, property key) | `grep` |

**Preference order for Java work: `lsp` → `grep` with `.java` glob → `view`.**

## LSP operation cheat sheet

All require `file` (absolute path) plus `line` + `character` (1-based) except `documentSymbol` (file only) and `workspaceSymbol` (query only).

```
lsp goToDefinition    file=src/main/java/.../UserService.java line=42 character=16
lsp findReferences    file=... line=... character=...  includeDeclaration=false
lsp hover             file=... line=... character=...
lsp documentSymbol    file=...
lsp workspaceSymbol   query=UserRepository          language=java
lsp goToImplementation file=... line=... character=...
lsp incomingCalls     file=... line=... character=...
lsp outgoingCalls     file=... line=... character=...
lsp rename            file=... line=... character=...  newName=findUser
```

## Gotchas

- **Indexing takes time.** JDTLS needs ~30–60s after opening a project to finish downloading Maven dependencies and building its index. `workspaceSymbol` returns empty during this window — wait, or fall back to grep briefly.
- **Must run `./mvnw compile` at least once** to populate `~/.m2/repository` with transitive deps; otherwise JDTLS can't resolve imports and produces false-positive errors.
- **No Lombok in this skill**, so none of the usual "JDTLS doesn't see `@Getter`-generated methods" problems apply.
- **One LSP server per language.** Don't run two Java servers (e.g. JDTLS + Sonar) in parallel — they fight for the workspace data dir.
- **Restart the LSP server** (`/lsp` in Copilot CLI, or reload window in VS Code) after large `pom.xml` edits to pick up new dependencies.
